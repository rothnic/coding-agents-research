# Near-Term Engineering Options (1–2 Weeks)

Date: 2026-05-24

## Context from Recent Activity

Recent commits on 2026-05-23 focused on maturing `local-agent-loop-v0` from v0.2 to v0.5 with:

- stronger schema and roadmap lifecycle behavior,
- optional worktree-backed execution,
- policy-driven local action adapters,
- richer validation command guardrails,
- expanded regression coverage.

In short: the core orchestration skeleton exists and now needs sharper proofs that it is reliable, observable, and extendable under realistic task mixes.

## Option 1 — Deterministic Replay + Failure Taxonomy (Reliability Proof)

### Goal
Prove that the loop behaves deterministically enough to debug and compare runs, especially across retries/failures.

### Scope (1–2 weeks)
- Add a run-level `session_id` and deterministic event ordering contract.
- Add a replay script that regenerates a normalized run summary from artifacts.
- Add failure taxonomy labels (e.g., `validation_failed`, `command_timeout`, `unsafe_path`, `allowlist_violation`) to task terminal events.
- Add regression fixtures for at least 5 representative failure modes.

### Proof outcomes
- Replaying the same fixture twice yields byte-equivalent normalized summaries.
- Every FAILED/BLOCKED terminal task has exactly one taxonomy code.
- A "failure dashboard" JSON can be generated from NDJSON artifacts with no manual triage.

### Why now
v0.5 introduced more execution branches (adapters + command policies); deterministic replay keeps complexity supportable.

## Option 2 — Multi-Task Scheduling Heuristics (Throughput/Unblocking Proof)

### Goal
Prove that lightweight scheduling policy improves throughput and reduces blocked-idle cycles versus FIFO/naive selection.

### Scope (1–2 weeks)
- Add pluggable task selection policies in worker/program handoff:
  - baseline FIFO,
  - dependency-aware oldest-unblocked,
  - "risk-first" (prioritize tasks with low retry count and known validators).
- Add synthetic queue fixtures with mixed OPEN/BLOCKED/retry-heavy tasks.
- Add benchmark harness that reports:
  - tasks completed per cycle,
  - cycles spent with no runnable tasks,
  - retry amplification.

### Proof outcomes
- Policy comparison report (JSON + markdown) generated by one command.
- At least one non-FIFO policy shows measurable benefit (e.g., +20% done tasks/cycle or fewer idle cycles) on synthetic fixtures.

### Why now
The current program loop already has trigger logic for unblocking/replanning; scheduling is the next highest leverage for practical performance.

## Option 3 — Adapter SDK + "Safe Custom Adapter" Pilot (Extensibility Proof)

### Goal
Prove the system can accept new local action adapters without weakening safety controls.

### Scope (1–2 weeks)
- Extract adapter interface contract (input validation, path safety, evidence capture, changed-file verification).
- Implement one new built-in adapter (e.g., `templated-file-batch`) that writes a constrained set of files from declarative inputs.
- Add adapter conformance tests shared across all adapters.
- Document "how to add an adapter" in spec docs with a checklist.

### Proof outcomes
- New adapter passes shared conformance suite with no special-case test hacks.
- Security invariants remain true across all adapters (allowlist, no shell execution, path bounds, dirty-worktree protection).
- Engineer can implement another adapter from docs within one day (dogfood metric).

### Why now
v0.5 established adapter direction; a stable SDK-like contract prevents future adapter drift.

## Option 4 — Operator UX: Explainability CLI + Run Health Checks (Adoption Proof)

### Goal
Prove that a human operator can quickly understand run status and intervene correctly.

### Scope (1–2 weeks)
- Extend `status` with concise explanations:
  - why replanning triggered,
  - why tasks are blocked,
  - what action is recommended next.
- Add `doctor` subcommand to validate artifacts and detect common misconfigurations.
- Add compact terminal table output (with JSON fallback).

### Proof outcomes
- Time-to-diagnose for seeded broken fixtures reduced (target: <5 minutes).
- `doctor` catches predefined misconfigurations with actionable messages.
- New operator can run validate → program → status/doctor workflow from README only.

### Why now
The engine is becoming capable; usability and diagnosability are now limiting factors for iteration speed.

## Recommended Sequence

1. **Option 1 first** (reliability/debug foundation).
2. **Option 3 second** (safe extensibility once replay/taxonomy exists).
3. **Option 2 or 4 third** depending on immediate priority:
   - pick **2** if optimization/throughput matters,
   - pick **4** if team adoption and faster triage matter.

## Suggested Definition of Done (for any option)

- One-command reproducible demo from a temp directory.
- Added/updated regression fixtures.
- Explicit success metrics in README/spec text.
- No reduction in existing test coverage for worker/program loop paths.
